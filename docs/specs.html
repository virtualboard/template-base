System Specification Catalog

Reusable blueprints for tech stack, local development, hosting, CI/CD, data, security, performance, and observability. Keep foundational documentation versioned, validated, and in lockstep with features.

8 templates Schema validated Automation ready

Why System Specs?

Feature specs capture user-facing value, but large projects also need durable references for architecture, tooling, and compliance. The specs/ directory gives every team a consistent starting point so onboarding, audits, and cross-team coordination stay predictable.

Cross-cutting Clarity

Document languages, deployment targets, access models, and safety policies once and link them from every feature touching that surface.

Best for: new markets, platform migrations, audit prep, or onboarding waves.

Automation Friendly

Each template ships with frontmatter validated by schemas/system-spec.schema.json. vb validate (or ./scripts/ftr-validate.sh) enforces metadata, ensuring docs stay parseable for agents and CI.

Frontmatter Schema

All templates use the same header so tooling can catalog ownership, lifecycle, and applicability.

Field Description Allowed Values
spec_type Template identifier (tech-stack, local-development, ...) One of 8 enums
title Human-readable spec title ≥ 3 characters
owner Team or role accountable for updates Free-form string
status Lifecycle state draft | approved | deprecated
last_updated ISO date or YYYY-MM-DD placeholder while templating YYYY-MM-DD
applicability Platforms covered (web, mobile, desktop, api, etc.) Array of kebab-case tokens
related_initiatives Optional project IDs/tags Array of alphanumeric/underscore values

Validation & Automation

CLI + Scripts

vb validate and ./scripts/ftr-validate.sh run schema checks for both feature specs and system specs. CI should execute these commands on every PR.

if command -v vb &> /dev/null; then
  vb validate
else
  ./scripts/ftr-validate.sh
fi

Automation Hooks

  • Frontmatter validated via schemas/system-spec.schema.json.
  • Arrays normalized (kebab-case tokens) for search/index pipelines.
  • Future automation can index these specs for agent prompts or UI catalogs.

Working with Specs

  1. Copy the relevant file from specs/ into docs/, a wiki, or feature branch folder.
  2. Update frontmatter fields (owner, status, applicability, last_updated) and remove placeholders.
  3. Fill in the Markdown sections with project context, diagrams, and links.
  4. Add references from feature specs (Implementation Notes + Links) so readers can find the canonical doc.
  5. Run vb validate (or ./scripts/ftr-validate.sh) before opening a PR.

Need a new template? Add it under specs/, extend schemas/system-spec.schema.json, and mention it here.
